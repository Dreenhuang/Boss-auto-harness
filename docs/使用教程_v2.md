## 一、项目简介

### 1.1 什么是Boss-auto-harness？

**Boss-auto-harness** 是一个基于 **Harness Engineering（AI编程工程化）** 理念构建的**全能型AI编程助手框架**。它将产品经理（PM）、开发者（Dev）、代码审查者（Reviewer）等多个角色整合到一个统一的自动化系统中，让AI能够像真正的产品经理一样思考，像资深工程师一样编码。

### 1.2 核心价值主张

```
传统AI编程的问题：
├── AI不问需求就开始写代码 → 做出来的东西不是你想要的
├── 写完的代码没有质量保证 → 充满bug和技术债
├── 文档和代码不同步 → 项目变成黑盒
├── 无法跨IDE使用 → 绑定单一工具
└── 缺乏工程化规范 → 输出质量参差不齐

Boss-auto-harness的解决方案：
├── ✅ 先问清楚再做 → 需求收集阶段充分沟通
├── ✅ 全自动质量门禁 → lint + test + review闭环
├── ✅ 文档-代码一致性 → 自动检测和同步
├── ✅ 跨6大IDE平台 → Trae/Cursor/Windsurf/VS Code/Claude Code/其他
└── ✅ Harness Engineering标准 → 企业级工程化能力
```

## 三、安装指南

### 3.1 系统要求

| 要求 | 最低版本 | 推荐版本 |
|------|---------|---------|
| 操作系统 | Windows 10 / macOS 12 / Ubuntu 20.04 | Windows 11 / macOS 14 / Ubuntu 22.04 |
| Node.js | 18.x | 20.x LTS |
| 包管理器 | bun / npm / pnpm | **bun** (推荐) |
| IDE | Trae IDE | Trae IDE (最佳) / Cursor / Windsurf |
| 内存 | 8GB RAM | 16GB RAM |
| 磁盘空间 | 500MB | 1GB |

### 3.2 全局安装（推荐）

#### Windows PowerShell

```powershell
# 方法1: 使用安装脚本（推荐）
cd boss-auto-harness
powershell -ExecutionPolicy Bypass -File install\install.ps1

# 方法2: 手动复制到全局目录
$source = "你的项目路径"
$target = "$env:USERPROFILE\.trae"

# 复制Skills
Copy-Item -Path "$source\skills" -Destination "$target\skills" -Recurse -Force

# 复制Agents
Copy-Item -Path "$source\trae-agents" -Destination "$target\agents" -Recurse -Force

# 复制主配置
Copy-Item -Path "$source\CLAUDE.md" -Destination "$target" -Force
```

#### Linux/macOS (Bash)

```bash
# 使用安装脚本
cd boss-auto-harness
chmod +x install/install.sh
./install/install.sh

# 或手动安装
SOURCE="./boss-auto-harness"
TARGET="$HOME/.trae"

cp -r $SOURCE/skills $TARGET/skills
cp -r $SOURCE/agents $TARGET/agents
cp $SOURCE/CLAUDE.md $TARGET/
```

### 3.3 项目级安装

如果只想在特定项目中使用：

```powershell
# 在目标项目中运行激活脚本
cd your-project
.\activate-in-project.ps1
```

这会创建符号链接指向全局安装，节省磁盘空间。

### 3.4 验证安装

```powershell
# 检查关键文件是否存在
Test-Path "$env:USERPROFILE\.trae\CLAUDE.md"      # 应该返回 True
Test-Path "$env:USERPROFILE\.trae\skills"         # 应该返回 True
Test-Path "$env:USERPROFILE\.trae\agents"         # 应该返回 True

# 查看已安装的Skills数量
(Get-ChildItem "$env:USERPROFILE\.trae\skills" -Directory).Count
# 应该显示 20+

# 查看版本信息
Get-Content "$env:USERPROFILE\.trae\rules\boss-auto-harness.rules" | Select-String "Version"
```

### 3.5 卸载

```powershell
# 运行卸载脚本
powershell -ExecutionPolicy Bypass -File "$env:USERPROFILE\.trae\uninstall-boss-auto-harness.ps1"
```

## 六、Agent协作机制

### 6.1 Agent角色介绍

| Agent名称 | 角色 | 职责 |
|----------|------|------|
| **Implementer** | 实现者 | 编码实现、编译验证、自检 |
| **Code-Reviewer** | 审查者 | 五维度审查、输出报告 |
| **Feedback-Observor** | 观察者 | 记录用户反馈和修正 |
| **Evolution-Runner** | 进化者 | 扫描反馈、生成优化建议 |
| **Agent-Orchestrator** | 编排器 | 智能调度、组合选择 |

### 6.2 Agent编排模式

根据项目复杂度，Orchestrator会自动选择最优模式：

| 模式 | 适用场景 | Agent组成 | 说明 |
|------|---------|----------|------|
| **Solo** | 小型低复杂度 | Boss直接执行 | 最简单快速 |
| **Dual-Core** | 中型中等复杂度 | Boss + Implementer | 常用模式 |
| **Triple-Core** | 中大型中高复杂度 | Boss + Implementer + Reviewer | 完整质保 |
| **Quad-Core** | 大型企业级 | 全量Agent + 并行实例 | 重型项目 |
| **Agile** | 快速迭代/Bug修复 | Boss一体化切换 | 灵活高效 |

### 6.3 Sub-Agent隔离原则

```
重要规则：每个Task必须使用fresh实例！

✅ 正确做法：
Task 1 → 创建 Implementer-A → 执行 → 销毁
Task 2 → 创建 Implementer-B → 执行 → 销毁
（即使同一个Agent定义，每次都是新实例）

❌ 错误做法：
Task 1 → 创建 Implementer → 执行
Task 2 → 复用同一个Implementer
（会导致Task 1的错误假设污染Task 2）
```

## 十、常见问题FAQ

### Q1: Boss模式和其他AI助手有什么区别？

**A**: 传统AI助手是"你说我做"，Boss模式是"我问你确认，然后我全自动做"。它更像一位真正的产品经理：先搞清楚需求，再安排开发，最后验收交付。而且它内置了完整的工程质量保障体系。

### Q2: 支持哪些编程语言和框架？

**A**: 理论上支持所有语言，但对以下有最佳优化：
- **前端**: React/Next.js/Vue/Nuxt/Svelte/Solid
- **后端**: Node.js/Python/Go/Rust/Java
- **移动端**: React Native/Flutter
- **桌面**: Electron/Tauri

它会自动检测项目技术栈并适配。

### Q3: 生成的代码质量如何？

**A**: 通过多层质量保证：
1. **Quality Gate**: lint + typecheck + 安全扫描
2. **Test Runner**: 单元测试 + 集成测试 + E2E测试
3. **Code Review**: 五维度专业审查
4. **Garbage Collector**: 技术债和熵增检测

实测代码质量可达中级以上工程师水平。

### Q4: 数据安全吗？

**A**: 
- 所有代码处理都在本地完成
- API Key存储在本地 memory/tokens.json
- 不会上传你的代码到外部服务器（除你主动推送Git）
- 市场调研使用WebSearch公开信息

### Q5: 可以自定义吗？

**A**: 完全可以！
- 使用 `/skill-builder` 创建新的Skill
- 修改 `CLAUDE.md` 调整规则
- 在 `rules/` 目录添加自定义规则
- 通过配置文件调整各种参数

### Q6: 遇到问题时怎么办？

**A**: 
1. 查看 `docs/` 目录下的日志和报告
2. 运行 `/bug-fixer` 自动诊断
3. 检查GitHub Issues看是否有类似问题
