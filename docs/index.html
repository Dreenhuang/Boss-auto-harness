一、基本概念

1.1 什么是 Boss-auto-harness？

Boss-auto-harness 是一套基于 TRAE IDE 的AI全自动开发模式技能包。它将产品经理的思维模式与全栈开发能力深度融合，通过结构化的 Skill 体系，实现从需求收集到部署上线的完整产品开发流程。

1.2 设计理念

🧠

Boss-harness 角色

资深产品经理兼全栈开发教练，负责引导用户完成产品开发的完整旅程

🤖

全自动开发

从模糊想法到可运行产品的端到端自动化流程

✅

2.2 Boss 模式（全自动开发模式）

Boss 模式是真正的全自动开发模式，整合 TRAE Autonomous Coding Agent 的执行能力。

📌 触发方式

输入: Boss、boss模式、boss moshi、Boss-harness、harness、全自动开发

核心特性

  • 问题跳过机制：支持 /跳过、/稍后问、/默认
  • 待确认清单追踪：自动记录跳过的问题，关键节点再次询问
  • AI 自主决策：🟡类问题3次不答→AI基于最佳实践决策
  • 竞品分析专项：自动搜索竞品+生成对比矩阵
  • 市场环境扫描：行业现状+趋势洞察+机会窗口识别
  • 用户画像构建：基础属性+行为特征+痛点挖掘
  • 决策记录机制：所有重要决策留痕+多源交叉验证+置信度标记

三、适用场景

适合的使用者

  • 独立开发者：想要快速将想法变为产品的开发者
  • 产品经理：需要全栈开发能力的产品专业人士
  • 创业者：需要快速验证产品原型的创业者
  • 学习者：想学习产品开发流程的初学者

适合的项目类型

  • Web 应用：React/Vue/Next.js 等前端项目
  • 后端服务：Node.js/Python/Go 等后端服务
  • 全栈项目：前后端分离或单体应用
  • 工具/库：CLI 工具、npm 包等
  • MVP 原型：快速验证想法的最小可行产品
⚠️ 不适合的场景
  • 大型企业级复杂系统（需要团队长期维护）
  • 高度定制化、特殊架构要求的项目
  • 已有成熟代码库的重构（需要人工评估）

六、高级功能与技巧

6.1 问题跳过机制

在 Boss 模式需求收集阶段，你可以随时跳过不想立即回答的问题：

Command 
# 跳过当前问题，继续下一个

# 加入待确认清单，关键节点会再次询问
/稍后问

# 接受 AI 推荐的默认值
/默认

6.2 AI 自主决策规则

🔴 类问题（必须）

强烈建议不跳过，坚持跳过则标记为"强制待确认"

🟡 类问题（建议）

3次模糊回答后AI自主决策，标注[AI决策]并加入待确认清单

⚪ 类问题（可选）

直接加入待确认清单，不阻塞流程

6.3 全链路记录系统

所有开发动作自动记录到 docs/ 目录，包含溯源四件套：

七、常见问题解答

Q1: Boss 模式和标准模式有什么区别？

Boss 模式下 AI 自主决策、自动 review + 自动修复、自动 commit。标准模式每步确认用户。

Q2: 如何暂停 Boss 模式？

输入 停、暂停 或 手动模式 即可切换回标准模式。

Q3: 如何查看已做出的决策？

输入 /决策 查看所有决策记录及其依据和置信度。

Q4: 图片生成有数量限制吗？

是的，MiniMax API 限制 50 张/日。上午最多 30 张，下午/晚上可用至 50 张。

Q5: 如何保证代码质量？

内置 Quality Gate、三层测试、五大维度 Code Review、自动 Bug Fixer 多重保障。

Q6: 如何自定义 Skills？

输入 /skill-builder 创建新 Skill，/evolution-engine 查看进化建议。

八、最佳实践

需求收集阶段

  • 诚实面对需求：不要夸大功能范围，MVP 优先
  • 明确目标用户：越具体越好，避免"所有人都是用户"
  • 区分必须和可选：🔴类问题尽量当场确认
  • 利用竞品分析：系统生成的竞品分析很有参考价值

开发阶段

  • 小步快跑：让 AI 按 Task 逐步开发，便于审查
  • 及时介入：发现方向偏差立即指出，不要等到 Phase 结束
  • 善用跳过：对不确定但非关键的问题使用 /跳过
  • 保留设计稿：有设计稿时务必发给 AI 参考
