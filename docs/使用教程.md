## 一、基本概念

### 1.1 什么是 Boss-auto-harness？

Boss-auto-harness 是一套基于 TRAE IDE 的**AI全自动开发模式技能包**。它将产品经理的思维模式与全栈开发能力深度融合，通过结构化的 Skill 体系，实现从需求收集到部署上线的完整产品开发流程。

### 1.2 设计理念

- **Boss-harness 角色**: 资深产品经理兼全栈开发教练，负责引导用户完成产品开发的完整旅程
- **全自动开发**: 从模糊想法到可运行产品的端到端自动化流程
- **质量保障**: 内置代码审查、测试验证、质量门禁等多重保障机制
- **可追溯性**: 全链路记录，所有决策和变更均可追溯

### 1.3 核心架构

```
用户想法 → 需求收集 → 设计规范 → 开发计划 → 代码实现 → 质量门禁 → 自动化测试 → 代码审查 → 构建发布 → 部署上线
```

### 2.2 Boss 模式（全自动开发模式）

Boss 模式是真正的全自动开发模式，整合 TRAE Autonomous Coding Agent 的执行能力。

**触发方式**:
- 输入: `Boss`、`boss模式`、`boss moshi`、`Boss-harness`、`harness`、`全自动开发`
- 关键字: `boss-mode`

**核心特性**:
- 问题跳过机制（支持 `/跳过`、`/稍后问`、`/默认`）
- 待确认清单追踪
- AI 自主决策（🟡类问题3次不答→AI基于最佳实践决策）
- 竞品分析专项
- 市场环境扫描
- 用户画像构建
- 决策记录机制

### 2.3 4 个智能体（Agents）

| 智能体 | 职责 | 调用时机 |
|-------|------|---------|
| code-reviewer | 审查代码 + 输出报告 | code-review 触发时 |
| implementer | 编码实现 + 编译验证 | dev-builder 复杂任务时 |
| feedback-observer | 记录用户反馈 | 用户给出修正意见时 |
| evolution-runner | 扫描反馈 + 生成进化建议 | Session 初始化时 |

## 三、适用场景

### 3.1 适合的使用者

- **独立开发者**: 想要快速将想法变为产品的开发者
- **产品经理**: 需要全栈开发能力的产品专业人士
- **创业者**: 需要快速验证产品原型的创业者
- **学习者**: 想学习产品开发流程的初学者

### 3.2 适合的项目类型

- **Web 应用**: React/Vue/Next.js 等前端项目
- **后端服务**: Node.js/Python/Go 等后端服务
- **全栈项目**: 前后端分离或单体应用
- **工具/库**: CLI 工具、npm 包等
- **MVP 原型**: 快速验证想法的最小可行产品

### 3.3 不适合的场景

- 大型企业级复杂系统（需要团队长期维护）
- 高度定制化、特殊架构要求的项目
- 已有成熟代码库的重构（需要人工评估）

## 五、详细操作步骤

### 5.1 阶段一：需求收集

#### 步骤 1: 启动需求收集

```
用户: 我想做一个任务管理应用
AI: 启动 product-spec-builder，开始需求收集流程...
```

#### 步骤 2: 竞品分析（Boss 模式自动执行）

系统自动搜索 3-5 个竞品，生成竞品分析矩阵，识别差异化机会。

#### 步骤 3: 市场扫描

分析行业现状、趋势洞察、机会窗口识别。

#### 步骤 4: 用户画像构建

收集目标用户基础属性、行为特征、痛点挖掘。

#### 步骤 5: 结构化需求确认

6 大维度 30+ 确认项（🔴必须/🟡建议/⚪可选），支持跳过机制。

#### 步骤 6: 生成产品需求文档

输出 `Product-Spec.md`，包含功能描述、用户故事、验收标准等。

### 5.2 阶段二：设计规范

```
用户: /design-brief-builder
AI: 生成 Design-Brief.md，定义视觉风格、组件规范、交互模式...
```

### 5.3 阶段三：开发计划

```
用户: /dev-planner
AI: 分析 Product-Spec.md，生成 DEV-PLAN.md，拆分 Phase 和 Task...
```

### 5.4 阶段四：项目开发

```
用户: /dev-builder
AI: 
1. 询问是否有设计稿
2. 按 DEV-PLAN.md 逐个 Phase 开发
3. 每个 Task 执行: 编码 → quality-gate → test → review → fix 循环
4. 所有 Task 完成后执行 Phase 级别验证
```

#### 开发循环详解

```
编码 (dev-builder)
  ↓
质量门禁 (quality-gate)
  ├── lint 检查
  ├── 类型检查 (tsc)
  ├── 安全扫描
  └── 复杂度检查
  ↓
自动化测试 (test-runner)
  ├── 单元测试
  ├── 集成测试
  └── E2E 测试
  ↓
代码审查 (code-review)
  ├── Stage 1: 需求匹配校验
  └── Stage 2: 代码质量检查
  ↓
通过 → 进入下一个 Task
不通过 → bug-fixer 修复 → 重新审查
```

### 5.5 阶段五：构建发布

```
用户: /release-builder
AI: 
1. 执行最终质量检查
2. 生成构建产物
3. 创建发布版本标签
4. 生成 CHANGELOG
```

### 5.6 阶段六：部署上线

```
用户: /deploy-engine
AI:
1. 识别项目类型，推荐部署目标
2. 执行部署脚本
3. 验证部署结果
4. 配置域名和 SSL（如需要）
```

### 5.7 阶段七：监控告警

```
用户: /monitor-setup
AI:
1. 配置 Sentry 错误追踪
2. 配置性能监控
3. 配置可用性监控
4. 配置日志管理
```

## 六、高级功能与技巧

### 6.1 问题跳过机制

在 Boss 模式需求收集阶段，你可以随时跳过不想立即回答的问题：

```
# 跳过当前问题，继续下一个
用户: /跳过

# 加入待确认清单，关键节点会再次询问
用户: /稍后问

# 接受 AI 推荐的默认值
用户: /默认
```

### 6.2 待确认清单追踪

系统会自动记录所有被跳过或待确认的问题，在关键节点（如 Phase 开始前、发布前）会重新询问。

```
# 查看当前待确认清单
用户: /待确认
AI: 📋 待确认清单 (3项)
    1. [🔴] 数据库选型 (Phase 3 开始前必须确认)
    2. [🟡] 支付集成方式 (AI 推荐: Stripe)
    3. [⚪] 第三方登录支持 (可选功能)
```

### 6.3 AI 自主决策规则

- **🔴类问题（必须）**: 强烈建议不跳过，坚持跳过则标记为"强制待确认"
- **🟡类问题（建议）**: 3次模糊回答后AI自主决策，标注[AI决策]并加入待确认清单
- **⚪类问题（可选）**: 直接加入待确认清单，不阻塞流程

### 6.4 Pipeline 编排（Boss 模式）

Pipeline 引擎自动将全流程串联为 DAG（有向无环图），按依赖关系执行：

```
Spec → Plan → Build → QualityGate → Test → Review → Deploy
```

**特殊能力**:
- 断点续跑: 从失败节点继续
- 并行执行: 无依赖节点同时执行

### 6.5 全链路记录系统

所有开发动作自动记录到 `docs/` 目录，包含溯源四件套：

```
【开发溯源信息】
AI大模型: [当前使用的模型]
开发IDE: [当前使用的IDE]
操作日期: YYYY-MM-DD
操作时间: HH:MM
```

## 七、常见问题解答

### Q1: Boss 模式和标准模式有什么区别？

| 维度 | 标准模式 | Boss 模式 |
|------|---------|----------|
| 决策 | 每步确认用户 | AI 自主决策 |
| 编码 | 主 Agent 执行 | TRAE Autonomous Agent |
| review | 展示报告等用户确认 | 自动 review + 自动修复 |
| commit | 用户确认后执行 | 自动 commit |
| 适用场景 | 需要精细控制 | 快速出活 |

### Q2: 如何让 Boss 模式暂停或切换回标准模式？

```
# 以下方式均可暂停 Boss 模式
用户: 停
用户: 暂停
用户: 手动模式
```

### Q3: 如果 AI 的开发方向不对，如何纠正？

```
用户: [指出具体问题]
AI: 调用 feedback-observer 记录反馈，调整开发方向
```

建议：在 code-review 阶段仔细查看 AI 的实现，发现问题立即指出。

### Q4: 如何查看项目当前进度？

Boss-harness 会自动检测项目状态并显示：

```
📊 项目进度检测
- Product Spec: [已完成/未完成]
- Design Brief: [已生成/未生成/未创建]
- DEV-PLAN: [已生成/未生成]
- 项目代码: [已创建/未创建]
当前阶段: [阶段名称]
下一步: [具体指令或操作]
```

### Q5: 图片生成有数量限制吗？

是的，MiniMax API 限制 50 张/日。上午（12点前）最多使用 30 张，下午/晚上可用至 50 张。

### Q6: 如何在其他 AI 编程软件中使用？

详见 [跨平台兼容方案](#跨平台兼容方案)。

### Q7: 开发过程中如何保证代码质量？

Boss-auto-harness 内置多重质量保障：

1. **Quality Gate**: 每次编码后自动执行 lint、typecheck、安全扫描
2. **三层测试**: 单元测试 → 集成测试 → E2E 测试
3. **Code Review**: 五大维度逐项校验（语法、规范、需求匹配、可维护性、安全性）
4. **Bug Fixer**: 发现问题自动修复，最多 5 次循环

## 八、最佳实践

### 8.1 需求收集阶段

1. **诚实面对需求**: 不要夸大功能范围，MVP 优先
2. **明确目标用户**: 越具体越好，避免"所有人都是用户"
3. **区分必须和可选**: 🔴类问题尽量当场确认
4. **利用竞品分析**: 系统生成的竞品分析很有参考价值

### 8.2 开发阶段

1. **小步快跑**: 让 AI 按 Task 逐步开发，便于审查
2. **及时介入**: 发现方向偏差立即指出，不要等到 Phase 结束
3. **善用跳过**: 对不确定但非关键的问题使用 `/跳过`，后期统一确认
4. **保留设计稿**: 有设计稿时务必发给 AI 参考

### 8.3 审查阶段

1. **仔细阅读 Review 报告**: 特别是需求匹配校验部分
2. **关注集成问题**: 跨 Task 的导入关系、命名一致性
3. **验证功能**: 启动项目手动测试关键流程

### 8.4 发布阶段

1. **完整测试**: 确保 E2E 测试全部通过
2. **检查敏感信息**: 确认没有密钥硬编码
3. **备份代码**: 发布前创建 Git 标签
4. **配置监控**: 部署后立即配置错误追踪

#### 方案二：Markdown 规则文件

CLAUDE.md 本身是纯 Markdown 格式，可直接复制到其他平台：

```

CLAUDE.md → .cursor/rules/boss-mode.md
CLAUDE.md → .windsurf/rules/boss-mode.md
CLAUDE.md → .github/copilot-instructions.md
```
