name: record-keeper
description: 项目开发全链路记录管家。每个开发、修改、评审、提交动作必须完整留痕。无合规记录禁止进入下一流水线节点。
---

[任务]
    为项目开发全链路的每一个动作生成结构化记录，确保全程可追溯、可复盘、可审计。
    所有记录统一归档至 docs/ 目录，按阶段分类存储。

**⚠️ 调用方式说明（重要）**：
本 Skill **不是独立调用的Skill**，而是**嵌入在各Skill内部的记录逻辑**：
- product-spec-builder 完成后 → **内部调用** record-keeper 逻辑生成需求记录
- dev-builder Task完成后 → **内部调用** record-keeper 逻辑生成变更记录
- code-review 完成后 → **内部调用** record-keeper 逻辑生成评审记录
- quality-gate / test-runner 完成后 → **内部调用** record-keeper 逻辑生成质量/测试记录
- git-committer 提交后 → **内部调用** record-keeper 逻辑生成版本记录
- pipeline-engine Node完成后 → **内部调用** record-keeper 逻辑生成流水线记录

**用户无需手动调用 /record-keeper**，系统自动触发。

[依赖检测]
    本 Skill 由其他 Skill 在关键节点自动调用，不由用户直接触发。

    **自动调用时机**：
    - product-spec-builder 生成 Spec 后 → 记录需求文档
    - dev-builder 每个 Task 完成后 → 记录代码变更
    - code-review 每次评审后 → 记录评审报告
    - quality-gate 执行后 → 记录质量检查结果
    - test-runner 执行后 → 记录测试结果
    - git-committer 提交后 → 记录 Git 提交信息
    - pipeline-engine 每个 Node 完成后 → 记录流水线状态

[第一性原则]
**无记录不流转**：任何环节未按要求生成标准记录，视为流程不达标，暂停流转。
**溯源必真实**：溯源信息必须据实填写，无法识别时标注「暂未识别」，禁止杜撰。
**历史不可篡改**：只允许新增、增量更新，禁止删除、覆盖原始记录。

[溯源信息强制规则]
**所有记录文件头部必须包含以下4项溯源信息，缺一不可**：

```markdown
【开发溯源信息】
AI大模型：[模型名称，如 GLM-5.1 / Claude-3.5-Sonnet]
开发IDE：[IDE名称，如 Trae IDE / VS Code]
操作日期：[YYYY-MM-DD]
操作时间：[HH:MM]
```

**溯源规则**：
- 每一轮修改、返工、二次更新，必须单独生成一条独立溯源记录
- 不得复用旧日期、旧时间、旧模型、旧 IDE 信息
- 无法识别时只能标注「暂未识别」，禁止杜撰
- 溯源信息缺失 = 无效记录，直接打回重写

[记录内容硬性规范]

**每条记录必须包含**：
1. 阶段说明（当前处于哪个开发阶段）
2. 工作内容（具体做了什么）
3. 关联任务编号（#T001, #P01 等）
4. 问题与解决方案（遇到的问题及如何解决）
5. 下一步动作（接下来要做什么）

**代码变更类记录必须精确到**：
1. 文件完整路径（如 src/components/Header.tsx）
2. 关键改动位置（行号或函数名）
3. 改动原因（为什么改）
4. 影响范围（影响了哪些功能/模块）

**需求变更类记录必须写明**：
1. 变更前内容
2. 变更后内容
3. 变更原因
4. 影响范围

**评审/报错/异常类记录必须写明**：
1. 问题根因
2. 整改标准
3. 后续规避方案

[流水线流转约束]
- 未完成当前环节规定记录 + 未补全溯源信息 → 禁止流转到下一节点
- 返工、评审驳回、流程中断 → 必须单独生成异常记录并留存溯源信息
- 每轮开发闭环结束 → 自动生成本轮开发汇总记录

[严格禁止行为]
- ❌ 省略记录、简化记录、敷衍填空、自由发挥改写模板
- ❌ 乱放记录文件、自建非规范目录与文档
- ❌ 不关联任务编号、缺时间戳、缺溯源信息的无效记录
- ❌ 跳过记录、跳过溯源直接编码、评审、提交代码
- ❌ 编造 AI 模型、IDE、日期时间等溯源信息

[初始化]
    检测 docs/ 目录是否存在，不存在则创建完整目录结构。
