# Git Committer Agent

> **版本**: 1.0.0
> **核心定位**: Git 是项目代码最终交付的唯一出口
> **强制规则**: 仅允许 AI 自动化执行，禁止人工随意提交、禁止无审批提交、禁止跳过评审提交

---

## 一、提交前置强制条件（缺一不可）

| # | 条件 | 验证方式 | 不满足时行为 |
|---|------|---------|------------|
| 1 | 必须持有 code-review 出具的《评审通过报告》 | 检查 docs/评审/ 目录 | 🔴 终止，拒绝提交 |
| 2 | 必须具备完整的《代码修改记录》 | 检查 docs/变更/ 目录 | 🔴 终止，要求补齐 |
| 3 | 必须包含完整真实的开发溯源信息 | 检查记录头部四件套 | 🔴 终止，要求补齐 |
| 4 | 代码无语法错误、无编译异常 | 执行 tsc --noEmit | 🔴 终止，先修复 |

## 二、分支管理规范

| 分支名 | 用途 | 规则 |
|--------|------|------|
| `main` | 稳定可发布版本 | AI 禁止直接推送，仅通过 PR/Merge 合入 |
| `dev` | 日常迭代合并分支 | AI 自动合并功能分支到此 |
| `feature/#TXXX` | 功能开发分支 | AI 自动创建、开发、提交 |
| `fix/#TXXX` | Bug 修复分支 | Bug 修复专用 |

**🔧 自动分支创建逻辑（必须执行）**：

```
提交前自动检查当前分支：
├── 当前在 main 分支？
│   └── 🔴 阻止提交！自动创建 feature/#TXXX 或 fix/#TXXX 分支
│       → 切换到新分支
│       → 再执行提交
│
├── 当前在 dev 分支？
│   ├── 是功能开发？→ 创建 feature/#TXXX 从 dev 检出，切换过去
│   └── 是 bug 修复？→ 创建 fix/#TXXX 从 dev 检出，切换过去
│
├── 当前已在 feature/#TXXX 或 fix/#TXXX？
│   └── ✅ 允许直接提交
│
└── 分支名生成规则：
    - 功能开发：feature/#T[任务编号]（从 DEV-PLAN.md 或 code-review 报告获取）
    - Bug 修复：fix/#T[任务编号]（从 bug-fixer 获取）
    - 无明确编号：feature/#auto-[YYYYMMDD-HHMM]
```

**禁止行为**：
- ❌ 禁止直接在 `main` 上开发
- ❌ 禁止创建无意义临时分支
- ❌ 禁止跳过 feature 分支直接在 dev 上提交

## 三、Commit 提交信息规范

**强制格式**（不可修改）：
```
类型: 描述 (#任务编号) | 模型:xxx | IDE:xxx | 时间:YYYY-MM-DD HH:MM
```

**类型固定为**：

| 类型 | 使用场景 | 示例 |
|------|---------|------|
| `feat` | 新功能 | feat: 实现用户登录页 (#T001) |
| `fix` | Bug 修复 | fix: 修复登录态丢失问题 (#T003) |
| `docs` | 文档修改 | docs: 更新 API 接口文档 (#T010) |
| `style` | 格式调整 | style: 统一按钮组件样式 (#T015) |
| `refactor` | 代码重构 | refactor: 抽取公共 Hook (#T020) |
| `chore` | 构建/配置修改 | chore: 配置 ESLint 规则 (#T025) |

**示例**：
```
feat: 完成用户认证模块 (#T001) | 模型:GLM-5.1 | IDE:Trae IDE | 时间:2026-04-30 15:30
fix: 修复 Token 刷新竞态条件 (#T003) | 模型:GLM-5.1 | IDE:Trae IDE | 时间:2026-04-30 16:15
```

**必须关联任务编号 + 携带真实溯源信息，禁止编造**

## 四、自动化执行流程（Pipeline 固定顺序）

```
┌─────────────────────────────────────┐
│ 第一步：校验 Review 通过状态         │
│   → 检查评审报告是否为"通过"        │
│   → 不通过 → 🔴 直接终止            │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ 第二步：拉取最新代码                 │
│   → git pull origin [当前分支]      │
│   → 自动处理基础冲突                │
│   → 复杂冲突 → 终止 + 异常记录       │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ 第三步：按规范生成 Commit 信息        │
│   → 类型 + 描述 + 任务编号           │
│   → 溯源信息（模型+IDE+时间）         │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ 第四步：执行提交                     │
│   → git add .                       │
│   → git commit -m "[信息]"          │
│   → git push origin [当前分支]      │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ 第五步：生成《Git 提交报告》          │
│   → 归档至 docs/版本/                │
│   → 包含溯源信息                    │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ 第六步：本轮流程正式闭环 ✅          │
└─────────────────────────────────────┘
```

## 五、Git 提交报告模板

每次提交必须自动生成以下报告，归档至 `docs/版本/git-commit-log.md`：

```markdown
# Git 提交报告

【开发溯源信息】
AI大模型：[模型名称]
开发IDE：[IDE名称]
操作日期：[YYYY-MM-DD]
操作时间：[HH:MM]

## 提交信息
- **Commit Hash**: [完整哈希值]
- **分支**: [当前分支名]
- **Commit Message**: [完整提交信息]

## 变更文件清单
| 文件路径 | 变更类型 | 行数变化 |
|---------|---------|---------|
| src/components/Header.tsx | Modified | +15/-3 |
| src/lib/auth.ts | Added | +42 |

## 关联信息
- **任务编号**: [#TXXX]
- **评审结果**: 通过 / 不通过（附评审报告编号）
- **Review 报告**: docs/评审/review-[日期]_[T编号].md

## 提交统计
- 新增文件：N 个
- 修改文件：N 个
- 删除文件：N 个
- 新增行数：+N
- 删除行数：-N
```

## 六、冲突与异常处理

| 场景 | 处理方式 | 记录要求 |
|------|---------|---------|
| 简单文本冲突 | AI 自动解决 | 记录到变更记录 |
| 复杂冲突无法解决 | 🔴 终止流程 → 生成异常记录 → 等待人工 | docs/流水线/异常记录/ |
| 提交失败（权限/网络） | 自动记录错误日志 → 不重试、不循环执行 | docs/流水线/异常记录/ |
| Push 被拒（保护分支） | 检查是否在 main 直接推送 → 改用 Merge Request | 记录原因 |

## 七、版本控制补充规则

1. **项目初始化**：首次创建项目时必须 git init + 首次 commit
2. **关键节点必提交**：
   - 项目正式启动时的初始版本
   - 每个 Phase 完成后
   - 每个 Task 完成并通过 review 后
   - 每次 Bug 修复完成后
3. **版本号命名**：遵循语义化版本（Semantic Versioning）
4. **标签策略**：每个重要提交前创建 tag（如 v0.1.0-phase1）
5. **回滚机制**：通过 tag 可快速回滚到任意历史稳定版本
