# 三明治推理结构优化指南 (Sandwich Reasoning Optimization Guide)

> **版本**: 1.0.0  
> **来源**: Prompt Engineering 最佳实践 + Chain-of-Thought + Structured Reasoning  
> **目标**: 提升 Boss-auto-harness 所有 Agent 的输出质量和一致性

---

## 一、什么是三明治推理结构？

### 1.1 核心概念

```
┌─────────────────────────────────────────────┐
│           🍞 上层：目标与约束                │
│   (Goal & Constraints Layer)                 │
│                                             │
│   • 明确的任务目标                           │
│   • 必须遵守的规则                           │
│   • 输出的质量标准                           │
│   • 成功的定义                               │
├─────────────────────────────────────────────┤
│           🥪 中层：推理与执行                │
│   (Reasoning & Execution Layer)             │
│                                             │
│   • 分步骤的思维链                           │
│   • 上下文信息的整合                         │
│   • 决策过程的记录                           │
│   • 中间结果的验证                           │
├─────────────────────────────────────────────┤
│           🍞 下层：输出与验证                │
│   (Output & Validation Layer)               │
│                                             │
│   • 标准化的输出格式                         │
│   • 质量检查清单                             │
│   • 自我反思与改进                           │
│   • 符合预期的确认                           │
└─────────────────────────────────────────────┘
```

### 1.2 为什么需要三明治结构？

```yaml
传统Prompt的问题：
  
  ❌ 模糊的目标 → AI不知道要做到什么程度
  ❌ 缺少约束 → 输出可能包含不相关内容
  ❌ 无结构化推理 → 思维跳跃、遗漏步骤
  ❌ 无输出规范 → 格式不一致、难以程序化处理
  ❌ 无自我验证 → 错误无法被发现
  
三明治结构的优势：
  
  ✅ 明确目标 → 输出精准匹配需求
  ✅ 清晰约束 → 避免无关内容和违规操作
  ✅ 结构化推理 → 完整的思维链，可追溯
  ✅ 标准输出 → 一致格式，便于集成和处理
  ✅ 自我验证 → 内置质量保证机制
```

## 三、推理增强技术

### 3.1 思维链 (Chain-of-Thought)

```yaml
basic_cot:
  description: "让AI展示思考过程"
  
  technique: "逐步推理"
  
  prompt_pattern: |
    让我们一步步思考这个问题：
    
    第一步：[首先分析什么]
    → 得出：[中间结论]
    
    第二步：[然后分析什么]
    → 得出：[中间结论]
    
    ...
    
    最后：[综合得出最终答案]
    
  effectiveness: 
    - 数学问题: +40% 准确率
    - 逻辑推理: +35% 准确率
    - 代码生成: +25% 正确率
    
advanced_cot:
  description: "带自我验证的思维链"
  
  technique: "推理 + 验证 + 修正"
  
  prompt_pattern: |
    ## 推理阶段
    [思维链过程]
    
    ## 验证阶段
    让我验证上面的推理：
    - 检查点1: [验证1]
    - 检查点2: [验证2]
    
    ## 发现问题？
    - 如果验证通过 → 确认答案
    - 如果发现问题 → 修正并重新推理
    
  effectiveness:
    - 减少幻觉: -60%
    - 提高准确性: +20%
```

### 3.2 自我反思 (Self-Reflection)

```yaml
reflection_technique:
  description: "让AI回顾和改进自己的输出"
  
  trigger_points:
    - 复杂任务完成后
    - 代码审查前
    - 最终输出前
    
  reflection_prompt: |
    在提交之前，让我重新审视我的工作：
    
    1. **目标回顾**: 我最初的目标是什么？达成了吗？
    2. **遗漏检查**: 我是否有遗漏的重要方面？
    3. **错误排查**: 是否有可能的错误或误解？
    4. **质量评估**: 这个输出达到专业水准了吗？
    5. **改进空间**: 如果再做一次，哪些地方可以更好？
    
    基于以上反思：
    - [需要修正的地方]
    - [保持不变的部分]
    - [最终确认的输出]
    
  benefits:
    - 捕获30%的自家错误
    - 提升输出质量15-25%
    - 展示专业性和严谨性
```

## 五、性能优化技巧

### 5.1 Token 效率优化

```yaml
token_optimization:
  
  problem: "长Prompt消耗更多Token，增加成本和延迟"
  
  solutions:
    
    use_variables:
      bad: "在每次Prompt中重复相同的规则说明"
      good: "使用变量引用，如 {coding_standards}"
      saving: "-40% Token (对于重复使用的规则)"
      
    layer_prompts:
      technique: "分层Prompt策略"
      description: "将规则分为'一次性设置'和'每次调用'"
      implementation:
        system_prompt: "固定规则（加载一次）"
        task_prompt: "具体任务（每次变化)"
      saving: "-60% Token (对于多次调用)"
        
    progressive_detail:
      technique: "渐进式详细信息"
      description: "先给摘要，需要时再展开"
      implementation:
        level_1: "任务概述 (必选)"
        level_2: "关键细节 (默认)"
        level_3: "完整背景 (按需)"
      saving: "-30% Token (对于简单任务)"
```

## 八、快速参考卡

### 8.1 三明治结构速查

```
┌─────────────────────────────────┐
│  🍞 上层: 要做什么 + 不能做什么  │  ← 10-20% 的Prompt
├─────────────────────────────────┤
│  🥪 中层: 怎么做 + 为什么这样做  │  ← 60-70% 的Prompt
├─────────────────────────────────┤
│  🍞 下层: 做好了吗 + 怎么证明    │  ← 10-20% 的Prompt
└─────────────────────────────────┘
```
