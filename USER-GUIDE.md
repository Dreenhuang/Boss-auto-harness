# Boss-auto-harness v2.0 用户手册

---

## 🎯 简介

Boss-auto-harness 是一款基于 Harness Engineering 方法论的 AI 编程工程化助手，旨在让 AI 编程更智能、更高效、更可靠。

### 核心价值
- 🤖 **全流程自动化**：从需求分析到部署上线，一键完成
- 📊 **质量保障**：内置质量门禁、代码审查、测试框架
- 🧹 **熵治理**：自动清理技术债，保持项目健康
- 🌐 **跨平台**：支持多种 IDE 和操作系统

---

## 🔧 核心功能

### 一、需求收集 (product-spec-builder)

**功能描述**：系统性收集项目需求，生成完整的产品规格文档

**使用方式**：
```bash
boss product-spec-builder
```

**核心特性**：
- 结构化需求确认清单
- 市场调研与竞品分析
- 用户画像构建
- 风险预判与缓解方案

**输出文件**：`Product-Spec.md`

---

### 二、开发计划 (dev-planner)

**功能描述**：将产品需求分解为可执行的开发任务

**使用方式**：
```bash
boss dev-planner
```

**核心特性**：
- Phase 阶段划分
- Task 任务拆解
- 依赖关系分析
- 工作量估算

**输出文件**：`DEV-PLAN.md`

---

### 三、代码开发 (dev-builder)

**功能描述**：自动执行代码开发，支持多种技术栈

**使用方式**：
```bash
boss dev-builder
```

**核心特性**：
- 多技术栈支持（React/Vue/Next.js 等）
- 代码质量监控
- 实时编译验证
- 自动修复机制

---

### 四、质量门禁 (quality-gate)

**功能描述**：自动化代码质量检查

**使用方式**：
```bash
boss quality-gate
```

**检查项**：
| 检查类型 | 说明 |
|---------|------|
| Lint | ESLint 代码规范检查 |
| TypeCheck | TypeScript 类型检查 |
| Security | 安全漏洞扫描 |
| Complexity | 代码复杂度分析 |

---

### 五、代码审查 (code-review)

**功能描述**：自动化代码审查，确保符合规范

**使用方式**：
```bash
boss code-review
```

**审查维度**：
1. 语法与运行合法性
2. 代码格式与工程规范
3. 需求匹配校验
4. 可维护性与代码质量
5. 健壮性与基础安全

---

### 六、测试执行 (test-runner)

**功能描述**：三层测试金字塔执行

**使用方式**：
```bash
boss test-runner
```

**测试层级**：
| 层级 | 说明 |
|------|------|
| 单元测试 | 代码级测试 |
| 集成测试 | 模块间集成 |
| E2E测试 | 端到端验证 |

---

### 七、一键发布 (release-builder)

**功能描述**：打包部署全流程自动化

**使用方式**：
```bash
boss release-builder
```

**支持目标**：
- Vercel 部署
- Docker 容器化
- VPS 服务器部署

---

### 八、图片生成 (image-generator)

**功能描述**：AI 图片生成，支持多种风格

**使用方式**：
```bash
boss image-generator --prompt "科技风格的产品封面" --size "landscape_16_9"
```

**参数说明**：
| 参数 | 说明 | 默认值 |
|------|------|-------|
| --prompt | 图片描述 | 必填 |
| --size | 图片尺寸 | square |
| --model | 生成模型 | minimax |

**支持尺寸**：
- `square` - 正方形
- `square_hd` - 高清正方形
- `portrait_4_3` - 竖版 4:3
- `portrait_16_9` - 竖版 16:9
- `landscape_4_3` - 横版 4:3
- `landscape_16_9` - 横版 16:9

## 🚀 Boss 模式详解

### 什么是 Boss 模式？

Boss 模式是一种**全自动开发模式**，AI 将：
1. 先进行全面的需求调研和问题澄清
2. 确认 PRD 后自主推进开发
3. 无需频繁打扰用户

### 触发方式

在 TRAE IDE 中输入以下任意关键字：
- `boss`
- `boss模式`
- `boss开发`
- `Boss-harness`
- `全自动开发`

### 执行流程

```
┌─────────────────────────────────────────────────────────┐
│ 第一阶段：需求收集与调研                                │
├─────────────────────────────────────────────────────────┤
│ 1. 市场调研 → docs/需求/市场扫描报告.md               │
│ 2. 竞品分析 → docs/需求/竞品分析报告.md               │
│ 3. 用户画像 → docs/需求/用户画像.md                   │
│ 4. 结构化需求确认（逐一向用户提问）                    │
│ 5. 技术风险预判                                       │
│ 6. PRD最终确认                                        │
├─────────────────────────────────────────────────────────┤
│ 第二阶段：规划与开发（全自动化）                        │
├─────────────────────────────────────────────────────────┤
│ 1. 生成开发计划 → DEV-PLAN.md                        │
│ 2. 自动执行编码（按 Phase → Task 顺序）               │
│ 3. 质量门禁验证                                       │
│ 4. 代码审查 → 修复循环                                │
│ 5. Git 自动提交                                       │
├─────────────────────────────────────────────────────────┤
│ 第三阶段：验收与交付                                   │
├─────────────────────────────────────────────────────────┤
│ 1. 功能验证                                           │
│ 2. 兼容性测试                                         │
│ 3. 部署上线                                           │
│ 4. 监控配置                                           │
└─────────────────────────────────────────────────────────┘
```

### 用户可用指令

| 指令 | 功能 |
|------|------|
| `/跳过` | 跳过当前问题 |
| `/稍后问` | 加入待确认清单 |
| `/默认` | 接受 AI 推荐的默认值 |
| `/待确认` | 查看待确认清单 |
| `/决策` | 查看已做出的决策 |
| `暂停` | 切换到手动模式 |

## 📊 Harness Engineering 六大支柱

### 1. 上下文架构
- 六维项目索引（元数据、目录结构、依赖关系、代码模式、关键实体、架构约束）
- 自动学习项目风格和规范

### 2. 架构约束
- 六层架构模型（Types → Config → Data → Service → Runtime → UI）
- 依赖分层检查，防止架构腐烂

### 3. 自验证循环
- Plan → Build → Verify → Fix 闭环
- 每阶段自动验证，确保质量

### 4. 上下文隔离
- 每个 Task 使用独立 Agent 实例
- 防止上下文污染

### 5. 熵治理
- 四维扫描引擎（文档一致性、架构违规、技术债、记录腐烂）
- 自动清理和维护

### 6. 可拆卸性
- Skill 模块化设计
- 支持多种 IDE 平台

## 🧪 最佳实践

### 项目初始化流程
```bash
# 1. 创建项目目录
mkdir my-project && cd my-project

# 2. 启动 Boss 模式
boss

# 3. 回答需求问题
# (AI 将自动完成后续工作)

# 4. 验证结果
ls -la
cat Product-Spec.md
cat DEV-PLAN.md
```

### 团队协作流程
```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   PM Agent   │────▶│ Coder Agent  │────▶│ Reviewer     │
│ 需求收集     │     │ 代码开发     │     │ 代码审查     │
└──────────────┘     └──────────────┘     └──────────────┘
                                               │
                                               ▼
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Git Committer│◀────│   Deploy    │◀────│   QA Agent   │
│  自动提交    │     │   部署上线   │     │  测试验证    │
└──────────────┘     └──────────────┘     └──────────────┘
```

### 性能优化建议
1. **增量开发**：将大需求拆分为多个 Phase
2. **缓存利用**：启用上下文缓存加速重复任务
3. **并行执行**：无依赖的 Task 可并行处理
4. **资源限制**：合理设置 API 调用配额

## 📈 版本控制

### Git 工作流
```
main (稳定版本)
    │
    └── dev (开发分支)
            │
            ├── feature/T001 (功能分支)
            ├── feature/T002 (功能分支)
            └── fix/T003 (修复分支)
```

### Commit 格式
```
类型: 描述 (#任务编号) | 模型:xxx | IDE:xxx | 时间:YYYY-MM-DD HH:MM

类型说明:
- feat: 新功能
- fix:  Bug 修复
- docs: 文档修改
- style: 格式调整
- refactor: 代码重构
- chore: 构建/配置修改
```

## 🔄 故障排除

### 日志查看
```bash
# 查看最近的执行日志
cat ~/.boss-harness/logs/latest.log

# 查看特定 Phase 的日志
cat ~/.boss-harness/logs/phase-01.log
```

### 常见错误

| 错误类型 | 原因 | 解决方案 |
|---------|------|---------|
| 权限不足 | PAT 权限不够 | 检查 PAT 权限配置 |
| 网络超时 | 网络问题 | 检查网络连接 |
| 依赖缺失 | 缺少依赖包 | 运行 `bun install` |
| 编译失败 | 代码错误 | 查看错误信息修复 |
| 配额用尽 | API 调用超限 | 等待配额重置 |
