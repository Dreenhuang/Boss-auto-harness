## 1. 快速上手

### 1.1 你的第一个Boss Mode项目

在TRAE IDE聊天框中输入：

```
boss 帮我做一个个人待办事项(Todo)应用
```

然后你只需要**回答AI提出的问题**即可！整个过程中AI会：

1. 🔍 自动搜索竞品和市场信息
2. ❓ 逐步向你确认需求细节
3. 📄 生成完整的产品规格说明书(PRD)
4. ✅ 等你确认PRD后，全自动编码实现
5. ✅ 自动测试、审查、修复、提交

### 1.2 最简启动方式

| 输入内容 | 效果 |
|---------|------|
| `boss` | 启动Boss Mode，等待你描述项目 |
| `boss模式` | 同上 |
| `boss开发` | 同上 |
| `boss-harness` | 同上 |
| `全自动开发` | 同上 |
| `帮我做一个XXX` | 直接启动并描述项目 |

### 1.3 核心原则

记住Boss Mode的三大原则：

```
🎯 原则一：先问清楚，再开发
    → AI会先问你很多问题，把需求搞清楚
    → 问题全部确认后生成PRD
    
🎯 原则二：PRD确认后，不再打扰
    → 你确认PRD后，AI进入自主开发模式
    → 不再频繁询问，除非遇到重大异常
    
🎯 原则三：全程可干预
    → 随时可以用指令控制流程
    → 可以暂停、跳过、修改、取消
```

## 3. 第一阶段：需求收集详解

### 3.1 步骤0：项目状态检测

AI首先扫描当前目录，判断项目处于哪个阶段：

| 检测结果 | 行为 |
|---------|------|
| 无Product-Spec.md | 执行步骤1-6（完整需求收集） |
| 有Spec但未确认(.prd-confirmed不存在) | 跳到步骤6（PRD确认） |
| 有Spec且已确认 | 直接进入第二阶段 |

### 3.2 步骤1：市场调研（强制执行）

AI会自动执行：

```
AI: "我将先进行市场调研，了解行业现状和趋势..."
     ↓
   WebSearch 搜索行业趋势
     ↓
   生成市场扫描报告 → docs/需求/市场扫描报告.md
     ↓
   向你展示关键发现
     ↓
   确认市场机会和定位
```

**你需要做的**：浏览AI展示的市场信息，确认方向是否正确。

### 3.3 步骤2：竞品分析（强制执行）

```
AI: "现在进行竞品分析，搜索3-5个主要竞品..."
     ↓
   WebSearch 搜索竞品
     ↓
   生成竞品对比矩阵：
   ┌──────────┬────────┬────────┬────────┬────────┐
   │ 维度     │ 竞品A  │ 竞品B  │ 竞品C  │ 我们   │
   ├──────────┼────────┼────────┼────────┼────────┤
   │ 核心功能 │ ...    │ ...    │ ...    │ ...    │
   │ 技术栈   │ ...    │ ...    │ ...    │ ...    │
   │ 定价     │ ...    │ ...    │ ...    │ ...    │
   │ 用户体验 │ ...    │ ...    │ ...    │ ...    │
   └──────────┴────────┴────────┴────────┴────────┘
     ↓
   差异化追问："基于以上分析，我们的差异化策略是？"
```

**你需要做的**：回答差异化策略相关问题。

### 3.4 步骤3：用户画像构建

AI会收集以下信息来构建目标用户画像：

| 类别 | 收集的信息 | 示例 |
|------|-----------|------|
| 基础属性 | 年龄、职业、地域、技术能力 | "25-35岁开发者" |
| 使用场景 | 何时用、怎么用、频率 | "每天使用2-3次" |
| 行为偏好 | 操作习惯、界面偏好 | "喜欢简洁风格" |
| 核心痛点 | 现有方案的不足 | "现有工具太复杂" |

### 3.5 步骤4：结构化需求确认（核心环节）

这是最关键的步骤！AI会**逐一**向你提问，共6大维度30+个确认项：

#### 🔴 必须回答的问题（影响项目成败）

**产品定位与目标**
- 这个产品的核心目标是什么？（一句话描述）
- 目标用户是谁？
- 成功指标是什么？（如何判断产品成功）

**功能需求**
- 核心功能有哪些？（列出MVP功能清单）
- 用户的主要使用流程是什么？
- 有哪些必须有、可以有、以后再加的功能？

**技术选型**
- 项目类型？(Web前端/Web全栈/Desktop/CLI/Library)
- 偏好的技术栈？（前端框架、后端语言、数据库等）
- 是否需要集成第三方服务？

#### 🟡 建议回答的问题（影响体验质量）

**性能与安全**
- 性能要求？（响应时间、并发量）
- 数据安全要求？（加密、权限、合规）
- 是否需要离线支持？

**设计与体验**
- 设计风格偏好？（简约/科技/商务/活泼）
- 是否需要暗色模式？
- 动效需求？（简单过渡/丰富动画/无动效）

#### ⚪ 可选回答的问题（锦上添花）

**发布与运营**
- 期望的发布时间？
- 部署方式偏好？（Vercel/Docker/VPS）
- 是否需要监控告警？

### 3.6 问题处理机制

在回答问题时，你可以随时使用以下指令：

| 指令 | 效果 | 使用场景 |
|------|------|---------|
| `/跳过` 或 `skip` | 跳过当前问题 | 不想现在回答 |
| `/稍后问` 或 `later` | 加入待确认清单 | 需要时间思考 |
| `/默认` | 使用AI推荐的默认值 | 对该问题无偏好 |

**注意**：
- 🔴类问题强烈建议不要跳过
- 🟡类问题3次不答，AI会基于最佳实践自动决策
- ⚪类问题直接加入待确认清单，不阻塞流程

### 3.7 步骤5：技术风险预判

AI会评估项目的技术风险：

```
风险清单示例：

├──────┼────────────────────┼────────┼────────────────┤
│ R01  │ 第三方API稳定性     │ 🟡中   │ 增加重试机制   │
│ R02  │ 数据量增长性能瓶颈  │ 🔴高   │ 分库分表预留   │
│ R03  │ 移动端兼容性       │ 🟢低   │ 响应式设计     │
└──────┴────────────────────┴────────┴────────────────┘
```

这是第一阶段的**关键门槛**！

AI会向你展示完整的PRD摘要：


## 4. 第二阶段：自主开发详解

### 4.1 Agent Orchestrator 初始化

PRD确认后，AI首先初始化多Agent协作系统：

```
🔄 正在分析项目特征，选择最优Agent协作模式...
    ↓
  分析项目：类型=Web全栈 | 规模=M | 复杂度=中 | 阶段=开发
    ↓
  决策结果：推荐 Dual-Core 模式（置信度 0.87）
    ↓
✅ Agent组合就绪！
  当前模式：Dual-Core
  参与Agent：Boss(PM) + Implementer(编码) + Code-Reviewer(按需)
```

### 4.2 开发计划生成

AI自动生成详细的开发计划（DEV-PLAN.md）：

```
DEV-PLAN.md 内容结构：

# 开发计划 - TodoApp

## Phase 1: 项目初始化
  T001: 创建Next.js项目结构
  T002: 配置Tailwind CSS
  T003: 设置项目基础布局
  
## Phase 2: 核心功能
  T004: 实现任务数据模型
  T005: 实现任务列表组件
  T006: 实现新增任务功能
  ...
  
## Phase 3: 增强功能
  ...

每个Task包含：
  - 任务描述
  - 验收标准
  - 预估复杂度
  - 依赖关系
```

### 4.3 逐Task开发循环

对每个Task，AI执行以下循环：

```
Task T004: 实现任务数据模型
    │
    ▼
┌─────────────────────────────────────┐
│ 1. 编码实现 (Implementer)           │
│    - 编写TypeScript类型定义         │
│    - 实现数据CRUD操作               │
│    - 添加必要的注释                 │
└─────────────────┬───────────────────┘
                  ▼
┌─────────────────────────────────────┐
│ 2. 质量门禁 (Quality Gate)          │
│    - ESLint检查                     │
│    - TypeScript编译                  │
│    - 安全扫描                       │
│    - 复杂度检测                     │
│    结果: ✅ 通过 / ❌ 阻断           │
└─────────────────┬───────────────────┘
                  ▼
┌─────────────────────────────────────┐
│ 3. 单元测试 (Test Runner)           │
│    - 运行单元测试                   │
│    结果: ✅ 通过 / ❌ 失败           │
└─────────────────┬───────────────────┘
                  ▼
┌─────────────────────────────────────┐
│ 4. 代码审查 (Code Reviewer)         │
│    Stage 1: Spec符合性检查          │
│    Stage 2: 代码质量五维审查        │

                  ▼

                  ▼

```

在整个开发过程中，**垃圾箱机制在后台静默运行**：

## 6. 高级功能

### 6.1 多Agent协作模式

Boss Mode会根据项目特征自动选择最优的Agent协作模式：

| 模式名 | 适用场景 | Agent组成 |
|--------|---------|----------|
| **Solo** | 小型低复杂度 | Boss直接执行 |
| **Dual-Core** | 中型中等复杂度 | Boss + Implementer |
| **Triple-Core** | 中大型高复杂度 | + Code-Reviewer |
| **Quad-Core** | 企业级大规模 | 全量Agent多实例 |
| **Agile** | 快速迭代/Bug修复 | Boss一体化切换 |

### 6.2 动态调整

开发过程中，Agent组合会动态调整：

```
场景：项目中途发现复杂度飙升

Before: Dual-Core (Boss + Implementer)
After:  Triple-Core (+ Code-Reviewer常驻)

触发条件：
  - Phase转换时
  - 复杂度变化时
  - 连续失败时
  - 时间压力时
```

### 6.3 进化引擎

系统内置进化引擎，会持续学习改进：

```
feedback-observer 记录你的反馈
    ↓
evolution-runner 扫描反馈
    ↓
生成进化建议（展示给你确认）
    ↓
你确认后自动优化Skill和规则
```

### 6.4 图片生成

需要UI素材时，可以使用AI图片生成：

```
用户: 帮我生成一个登录页面的背景图
AI: 正在调用MiniMax API生成图片...
    ↓
  图片已保存到 assets/login-bg.png
```

**配额限制**：每日50张（上午最多30张）

## 7. 常用指令速查

### 7.1 触发Boss Mode

| 指令 | 说明 |
|------|------|
| `boss` | 启动Boss Mode |
| `boss模式` | 同上 |
| `boss开发` | 同上 |
| `boss-harness` | 同上 |
| `全自动开发` | 同上 |
| `帮我做一个XXX` | 直接启动+描述项目 |

### 7.2 流程控制指令

| 指令 | 说明 | 使用时机 |
|------|------|---------|
| `暂停` / `停止` | 暂停当前任务 | 需要介入时 |
| `继续` / `resume` | 恢复执行 | 处理完暂停原因后 |
| `取消` / `中止` | 终止当前任务 | 要放弃时 |
| `回到上一步` | 回退到上一节点 | 对结果不满意 |

### 7.3 问题处理指令

| 指令 | 说明 | 使用时机 |
|------|------|---------|
| `/跳过` 或 `skip` | 跳过当前问题 | 不想回答时 |

## 9. 故障排除

### 9.1 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| AI不问我直接开始开发 | 未识别到触发词或CLAUDE.md未加载 | 检查CLAUDE.md位置，重启IDE |
| 市场调研/竞品分析被跳过 | 规则未正确加载 | 确认使用v2.1+版本 |
| PRD确认后还在频繁提问 | 待确认清单中有🔴项 | 检查并确认阻塞项 |
| 代码质量差 | Quality Gate未正常执行 | 手动触发 `/quality-gate` |
| Git提交失败 | Token过期或权限不足 | 检查remote配置 |
| 图片生成失败 | API Key无效或配额用完 | 检查MINIMAX_API_KEY |

### 9.2 日志位置

如果出现问题，可以查看以下日志：

```
项目根目录/
```
